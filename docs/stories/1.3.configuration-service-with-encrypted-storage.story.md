# Story 1.3: Configuration Service with Encrypted Storage

## Status

## Story

**As a** user,
**I want** my API keys stored securely and encrypted,
**so that** my credentials are protected.

## Acceptance Criteria

1. `ConfigService` class with methods: `initialize`, `set`, `get`, `has`, `delete`
2. Uses `conf` library with encryption in OS-specific locations
3. Config schema defined with TypeScript interface
4. API keys never logged (redaction in place)
5. Environment variable support: `TM_PROVIDER`, `TM_API_KEY`
6. Unit tests cover storage/retrieval, environment precedence

> **Note:** `docs/prd/technical-assumptions.md` lists env vars as `TMR_PROVIDER` / `TMR_API_KEY`, while the epic AC specifies `TM_PROVIDER` / `TM_API_KEY`. Following the same precedence rule as the binary name (`tm` not `tmr`): **use `TM_PROVIDER` and `TM_API_KEY`** as the canonical names.

### `conf` Library — Encrypted Storage

`conf` stores config in OS-specific user data directories automatically:

```
macOS:   ~/Library/Application Support/@marlonvidal-tech-leadership-os/
Linux:   ~/.config/@marlonvidal-tech-leadership-os/
Windows: %APPDATA%\@marlonvidal-tech-leadership-os\
```

Encryption is enabled by passing `encryptionKey` to the constructor (AES-256):

```typescript
import Conf from 'conf';
import type { AppConfig } from '../types/config.types.js';

const store = new Conf<AppConfig>({
  projectName: '@marlonvidal/tech-manager-os',
  encryptionKey: 'tm-config-v1', // app-scoped obfuscation key
});
```

> The `encryptionKey` is embedded in the binary — this is "better than plaintext" protection for config files at rest, not cryptographically secret. This is the standard `conf` pattern for CLI tools.

[Source: docs/prd/technical-assumptions.md#security--configuration]

### Environment Variable Precedence

`get()` must check env vars first, then fall back to the conf store. The precedence order:

```
TM_PROVIDER env var  >  conf store value
TM_API_KEY  env var  >  conf store value
```

`initialize()` must call `dotenv.config()` early so `.env` files are loaded before any `get()` call.

```typescript
import dotenv from 'dotenv';

initialize(): void {
  dotenv.config(); // loads .env if present
  // store is already available via Conf constructor
}
```

[Source: docs/prd/technical-assumptions.md#security--configuration]
[Source: docs/prd/epics/epic-1-foundation-cli-infrastructure.md#story-13]

### API Key Redaction

API keys must never appear in logs or error output. A standalone utility is preferred over inline masking:

```typescript
// src/utils/redact.ts
export function redact(value: string): string {
  if (value.length <= 4) return '****';
  return `${'*'.repeat(value.length - 4)}${value.slice(-4)}`;
}
```

All paths inside `ConfigService` that could output the apiKey value (e.g., error messages, debug) must call `redact()`.

[Source: docs/prd/epics/epic-1-foundation-cli-infrastructure.md#story-13-ac4]

### Mocking `conf` in Tests

Do NOT let tests write to real OS config directories. Mock the `conf` module:

```typescript
import { jest } from '@jest/globals';

jest.unstable_mockModule('conf', () => {
  const store = new Map<string, unknown>();
  return {
    default: jest.fn().mockImplementation(() => ({
      get: (key: string) => store.get(key),
      set: (key: string, value: unknown) => store.set(key, value),
      has: (key: string) => store.has(key),
      delete: (key: string) => store.delete(key),
    })),
  };
});
```

Use `jest.unstable_mockModule` (ESM-compatible) instead of `jest.mock` for ESM modules.
Reset mock store state in `beforeEach` to avoid test pollution.
Restore env vars modified in tests using `afterEach` cleanup.
