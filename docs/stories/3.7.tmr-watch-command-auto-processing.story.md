# Story 3.7: `tmr watch` Command for Auto-Processing

## Status

Done

## Story

**As a** leader,
**I want** inbox to be processed automatically when files are added,
**so that** I don't have to remember to run process.

## Acceptance Criteria

1. `tmr watch` monitors `inbox/` directory
2. Uses `chokidar` for file watching
3. Debounces multiple file additions (waits 5 seconds)
4. Runs `tmr process` automatically
5. Displays activity log
6. Can be stopped with Ctrl+C
7. Stores PID in `.system/watch.pid`
8. Integration test validates file watching

### WatchService Implementation Pattern

```typescript
// src/services/watch.service.ts
import chokidar from 'chokidar';
import path from 'path';

export interface WatchOptions {
  verbose: boolean;
  plain: boolean;
}

export class WatchService {
  constructor(
    private readonly _processSvc: InboxProcessService,
    private readonly _fs: FileSystemService,
  ) {}

  async start(workspaceRoot: string, opts: WatchOptions): Promise<void> {
    // 1. Ensure .system/ dir, write PID
    // 2. chokidar.watch inbox/ — on 'add': debounce 5s then call _processSvc.run()
    // 3. SIGINT/SIGTERM: watcher.close(), remove PID, exit
  }
}
```

**Debounce pattern:**

```typescript
let debounceTimer: NodeJS.Timeout | undefined;
watcher.on('add', (filePath) => {
  // log file detected
  if (debounceTimer) clearTimeout(debounceTimer);
  debounceTimer = setTimeout(async () => {
    // run process
    await this._processSvc.run(workspaceRoot, { ...opts, dryRun: false });
  }, 5000);
});
```

**PID file:** `path.join(workspaceRoot, '.system', 'watch.pid')` — write `String(process.pid)`.

### Completion Notes List

- `WatchService.start()` uses an `EventEmitter` internally to bridge the SIGINT/SIGTERM signal handlers to the blocking `await new Promise(…)` — this keeps the function awaitable while the watcher runs, and delegates `process.exit(0)` to the caller after cleanup rather than inside the signal callback, which prevents unhandled rejection in tests.
- `_watcherFactory` injectable option added to `WatchOptions` so integration tests can inject a mock watcher without hitting OS file-descriptor limits (chokidar EMFILE errors in heavy test environments).
- `debounceMs` injectable option allows integration tests to use 50–200ms debounce instead of the 5-second production value.
- `buildInboxProcessService()` duplicated verbatim from `process.command.ts` into `watch.command.ts` — acceptable for now; could be extracted to a shared factory if a third command needs it.
- CLI test updated: replaced the `"'watch' coming soon"` assertion with a command-name/description registration check.
- 16 new tests added (5 command unit + 11 integration); total suite: 730 passing.
- apply-qa-fixes (v1.3): Gate was PASS with no top_issues, no coverage gaps, no NFR failures. QA's two future recommendations (extract factory, verbose extension logging) are low-priority and deferred. No code changes required; status advanced to Done.
- apply-qa-fixes (v1.4): Second QA review (2026-04-14) identified a flaky integration test — `logs file name when a watched file is added` failing under full suite load due to race condition. `watchSvc.start()` performs real FS I/O (`createDirectory`, `writeFile`) before registering the chokidar `on('add', ...)` listener; 50ms setup margin was insufficient under load. ### Code Quality Assessment

Implementation is clean and well-structured. `WatchService` is properly dependency-injected, uses `Result<T>` from `InboxProcessService`, and handles both signal types gracefully. The injectable `_watcherFactory` and `debounceMs` options are a thoughtful design that keeps tests fast and deterministic without compromising production behaviour.

### Refactoring Performed

- **File**: `src/services/watch.service.ts`
  - **Change**: `process.on` → `process.once` for SIGINT and SIGTERM handlers; added `cleaning` boolean guard.
  - **Why**: `process.on` would accumulate duplicate handlers if `start()` were ever called multiple times, or if both SIGINT and SIGTERM fired simultaneously.
  - **How**: `process.once` auto-removes the handler after first invocation; `cleaning` flag makes `cleanup()` idempotent against concurrent signals.

- **File**: `tests/services/watch.service.integration.test.ts`
  - **Change**: Updated spy target from `process.on` to `process.once` to match refactored service.
  - **Why**: Spy must intercept the same method the service calls.
  - **How**: Functional equivalence preserved; all 16 tests still pass.
