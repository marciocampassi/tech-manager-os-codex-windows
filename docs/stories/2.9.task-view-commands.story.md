# Story 2.9: Time-Based Task View Commands

## Status

Draft

## Story

**As a** manager,
**I want** CLI commands to view my tasks organized by time horizon (`tmr today`, `tmr this-week`, `tmr this-month`, `tmr this-quarter`),
**so that** I can quickly see what needs my attention now, this week, and in the longer term without opening Obsidian.

---

## Acceptance Criteria

### 1. `tmr today` command

- Reads `my-tasks/today.md` from the configured workspace root
- Displays content with formatted, color-coded terminal output:
  - Header: `üìÖ Today ‚Äî {date}` (chalk bold)
  - If file is empty or missing: `No tasks for today. Run \`tmr process\` to populate from inbox.`
  - Content displayed as-is (markdown rendered inline using `marked-terminal` or plain fallback)
- Optional flags:
  - `--plain` ‚Äî disables colors, outputs plain text (suitable for piping)
  - `--json` ‚Äî outputs raw file content as JSON `{ "period": "today", "content": "..." }`

### 2. `tmr this-week` command

- Reads `my-tasks/this-week.md` from the workspace root
- Same display format as `tmr today`, header: `üìÜ This Week ‚Äî {week range}`
- Same `--plain` and `--json` flags

### 3. `tmr this-month` command

- Reads `my-tasks/this-month.md` from the workspace root
- Header: `üóìÔ∏è  This Month ‚Äî {month name} {year}`
- Same `--plain` and `--json` flags

### 4. `tmr this-quarter` command

- Reads `my-tasks/this-quarter.md` from the workspace root
- Header: `üìä This Quarter ‚Äî Q{N} {year}`
- Same `--plain` and `--json` flags

### 5. Task file initialization during `tmr init`

- `tmr init` creates all four task files with a default empty template:
  ```markdown
  # Tasks ‚Äî {period label}

  _Run `tmr process` to populate this file from your inbox._
  ```
- Files created: `my-tasks/today.md`, `my-tasks/this-week.md`, `my-tasks/this-month.md`, `my-tasks/this-quarter.md`
- The `my-tasks/` directory is already created by `workspace-builder.ts`; this story adds the **file content** creation
- Idempotent: if files already exist, skip without error

### 6. `TaskViewService` class

- `src/services/task-view.service.ts`:
  - `readTaskFile(period: TaskPeriod, workspaceRoot: string): Promise<string>` ‚Äî reads and returns file content
  - `formatHeader(period: TaskPeriod): string` ‚Äî returns the colored header string
  - `renderContent(content: string, plain: boolean): string` ‚Äî renders markdown for terminal
- `TaskPeriod` type: `'today' | 'this-week' | 'this-month' | 'this-quarter'`
- Period ‚Üí file path map:
  | Period | File |
  |--------|------|
  | `today` | `my-tasks/today.md` |
  | `this-week` | `my-tasks/this-week.md` |
  | `this-month` | `my-tasks/this-month.md` |
  | `this-quarter` | `my-tasks/this-quarter.md` |

### 7. Commands registered in `src/cli.ts`

- All four commands registered at the top-level CLI (not as subcommands):
  ```
  tmr today [--plain] [--json]
  tmr this-week [--plain] [--json]
  tmr this-month [--plain] [--json]
  tmr this-quarter [--plain] [--json]
  ```
- Alternatively: single `tmr task <period>` command with `period` as positional argument (see Dev Notes ‚Äî Command Design Decision)

### 8. Unit tests

- `TaskViewService`:
  - `readTaskFile`: success path, file not found (returns empty template message)
  - `formatHeader`: correct labels for all 4 periods
  - `renderContent`: plain mode vs color mode
- Command unit tests:
  - Each period command invokes `TaskViewService` with correct period
  - `--plain` flag passes through
  - `--json` flag produces JSON output
  - Error when workspace not initialized (no config)

### 9. Integration test

- `tmr today` on a workspace with populated `today.md` ‚Üí displays correct content
- `tmr today` on workspace with missing `today.md` ‚Üí displays empty state message
- `tmr this-week --plain` ‚Üí outputs plain text without ANSI codes
- `tmr init` creates all four task files with default content

## Dev Notes

### Command Design Decision

**Option A: Four separate top-level commands** (`tmr today`, `tmr this-week`, etc.)
- Pros: Most ergonomic, mirrors user mental model, shortest to type
- Cons: Uses 4 top-level command slots

**Option B: Single `tmr task <period>` subcommand** (`tmr task today`, `tmr task this-week`)
- Pros: Namespaced, leaves top-level clean
- Cons: One extra word to type, less intuitive

**Recommendation: Option A** ‚Äî the time-based views are first-class commands frequently used by managers. FR7 explicitly documents them as standalone commands. Implement as `tmr today`, `tmr this-week`, `tmr this-month`, `tmr this-quarter`.

### Header Formatting

```typescript
function formatHeader(period: TaskPeriod): string {
  const now = new Date();
  switch (period) {
    case 'today':
      return chalk.bold(`üìÖ Today ‚Äî ${format(now, 'EEEE, MMMM d, yyyy')}`);
    case 'this-week': {
      const start = startOfWeek(now, { weekStartsOn: 1 });
      const end = endOfWeek(now, { weekStartsOn: 1 });
      return chalk.bold(`üìÜ This Week ‚Äî ${format(start, 'MMM d')}‚Äì${format(end, 'MMM d, yyyy')}`);
    }
    case 'this-month':
      return chalk.bold(`üóìÔ∏è  This Month ‚Äî ${format(now, 'MMMM yyyy')}`);
    case 'this-quarter': {
      const q = Math.ceil((now.getMonth() + 1) / 3);
      return chalk.bold(`üìä This Quarter ‚Äî Q${q} ${now.getFullYear()}`);
    }
  }
}
```

Use `date-fns` for date formatting (already a common transitive dependency; add explicitly if not present).

### Task File Default Template

Generated during `tmr init` for each period:

```typescript
function generateTaskFileTemplate(period: TaskPeriod): string {
  const label = {
    'today': 'Today',
    'this-week': 'This Week',
    'this-month': 'This Month',
    'this-quarter': 'This Quarter',
  }[period];
  return `# Tasks ‚Äî ${label}\n\n_Run \`tmr process\` to populate this file from your inbox._\n`;
}
```

### Relationship with Epic 3

The `my-tasks/{period}.md` files are **written by** Epic 3 (Story 3.4: Task Extraction and Timeline Classification). This story (2.9) provides:

Epic 3 is the write layer; Epic 2 Story 2.9 is the read layer. Both are required for the full workflow.
