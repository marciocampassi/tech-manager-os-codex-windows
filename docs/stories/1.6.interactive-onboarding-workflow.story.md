# Story 1.6: Interactive Onboarding Workflow (`tm init`)

## Status

Done

## Story

**As a** new user,
**I want** an interactive setup wizard that guides me through configuration,
**so that** I can start using the system immediately without manual file editing.

## Acceptance Criteria

1. Displays branded welcome message
2. Interactive AI provider selection (inquirer)
3. API key input (password-style, hidden)
4. Connection validation with spinner
5. Collects manager profile:
   - Name, email, role, experience years
   - Management style
   - Strengths and development areas
6. Collects career goals (short-term, long-term, target role)
7. Collects leadership context (manager's name, email, expectations)
8. Creates complete directory structure
9. Generates initial files: `my-career/profile.md`, `my-career/pdp.md`, `my-leadership/profile.md`
10. Creates IDE integration files (`.cursor/rules/tmr/`, `.claude/agents/`, `.gemini/agents/`)
11. Displays next steps on completion
12. Integration test validates full workflow end-to-end

## Dev Notes

### Previous Story Insights (from Stories 1.3‚Äì1.5)

- **ESM is active**: all source imports need `.js` extensions; test imports resolved by `moduleNameMapper` in Jest config so `.js` works in tests too.
- **`jest.unstable_mockModule` is required for ESM mocks** ‚Äî `jest.mock` does NOT work with ESM. Always declare mocks before any `await import()` of the module under test.
- **Dynamic import pattern**: after `jest.unstable_mockModule(...)`, use `const { InitCommand } = await import('../../src/commands/init.command.js')` ‚Äî NOT static import.
- **Typed mock functions**: declare as `jest.fn<() => Promise<T>>()` to avoid TypeScript inferring `never` on `.mockResolvedValue(...)` calls.
- **`Object.setPrototypeOf` on custom errors**: already applied in `AIProviderError` and `FileSystemError`; no new error classes in this story.
- **Singleton side effects**: `configService` is a module-level singleton backed by `conf` ‚Äî mock the whole module with `jest.unstable_mockModule` before importing `init.command.js` to avoid writing to real OS config during tests.

[Source: docs/stories/1.5.file-system-service.story.md#previous-story-insights]

### Tilde Expansion for Workspace Path

`~/tech-leadership-workspace` must be converted to an absolute path before any `fs-extra` call:

```typescript
import { homedir } from 'node:os';
import { join } from 'node:path';

function expandPath(p: string): string {
  return p.startsWith('~/') ? join(homedir(), p.slice(2)) : p;
}
```

Place `expandPath` in `onboarding.prompts.ts` and call it inside `promptWorkspacePath()` before returning.

### Mocking `inquirer` in Tests

Use `.mockResolvedValueOnce()` chaining to supply one answer object per `inquirer.prompt()` call in the order they are invoked during `InitCommand.run()`:

```typescript
const mockPrompt = jest.fn<() => Promise<Record<string, unknown>>>();

jest.unstable_mockModule('inquirer', () => ({
  default: { prompt: mockPrompt },
}));

// In each test:
mockPrompt
  .mockResolvedValueOnce({ workspacePath: '/tmp/test-ws' })     // promptWorkspacePath
  .mockResolvedValueOnce({ provider: 'openai' })                // promptProviderSelection
  .mockResolvedValueOnce({ apiKey: 'sk-test' })                 // promptApiKey

    role: 'EM', experienceYears: 5,
    managementStyle: 'Coaching',

    developmentAreas: 'Delegation' })
```

### `Promise.all` for Directory Creation

Creating all workspace directories serially is wasteful. Use `Promise.all` in `workspace-builder.ts`:

```typescript
const dirs = [
  'inbox', 'my-career', 'my-leadership', 'my-teams',
  // ... full list
];
await Promise.all(
  dirs.map(d => fileSystemService.createDirectory(path.join(workspacePath, d)))
);
```

The `FileSystemService.createDirectory` wraps `fs.ensureDir` which is safe to call concurrently for non-overlapping paths.
