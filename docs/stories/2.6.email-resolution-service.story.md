# Story 2.6: Email Resolution Service

## Status

## Story

**As a** developer,
**I want** a centralized service for hierarchical email resolution,
**so that** all commands consistently locate or create email profiles without duplicated logic.

## Acceptance Criteria

1. **`EmailResolutionService` Class:**
   - Method: `resolve(email: string, ws: string): Promise<IEntityLocation>`
   - Returns: `{ type: 'team' | 'leadership' | 'relationship', absolutePath: string, created: boolean }`
   - Hierarchy order:
     1. Check `/my-teams/_members/{email}/{email}.md` → type: `'team'`
     2. Check `/my-leadership/{email}/{email}.md` → type: `'leadership'`
     3. Check `/my-company/relationships/{email}/{email}.md` → type: `'relationship'`
     4. If not found, call `RelationshipService.addRelationship(email, {}, ws)` and return relationship path with `created: true`
   - Email is normalized to lowercase before all checks
   - Unit tests cover: team hit, leadership hit, relationship hit, auto-create

2. **Email Validation:**
   - Public method `validateEmail(email: string): boolean`
   - Returns `true` for valid format, `false` for empty/malformed/special-char-only strings
   - `resolve()` throws `Error` (message: `"Invalid email address: '{email}'"`) when `validateEmail` returns `false`
   - Unit tests cover: valid emails, empty string, malformed, no-at-sign

3. **Path Generator (Sync):**
   - Method: `generateWikiLink(email: string, resolvedPath: string, fromPath: string): string`
   - Computes relative path using `path.relative(path.dirname(fromPath), resolvedPath)`
   - Normalizes path separators to `/` (cross-platform)
   - Returns Obsidian wiki-link string: `[[{relPath}|{email}]]` (no `- ` prefix — callers add their own)
   - Example: `fromPath=/workspace/my-projects/platform/platform-composition.md`, `resolvedPath=/workspace/my-teams/_members/user@example.com/user@example.com.md` → `[[../../my-teams/_members/user@example.com/user@example.com.md|user@example.com]]`
   - Unit tests cover: team path, leadership path, relationship path, different nesting depths

4. **Integration with `ProjectService`:**
   - `ProjectService` constructor updated to accept `EmailResolutionService` as 4th dependency
   - Private `resolveEmailLocation()` method **removed** from `ProjectService`
   - `linkMember()`, `linkStakeholder()`, `linkMembers()`, `linkStakeholders()` updated to call `this._emailResolution.resolve()` and `this._emailResolution.generateWikiLink()`
   - Module-level singleton updated: `export const projectService = new ProjectService(fileSystemService, templateService, relationshipService, emailResolutionService)`
   - `IEmailLocation` import removed from `project.service.ts` (no longer needed internally)
   - `ProjectService` unit tests updated: add `MockEmailResolution` mock; resolution-hierarchy tests moved to `email-resolution.service.test.ts`

5. **Performance Caching:**
   - In-memory `Map<string, IEntityLocation>` cache keyed by `"${email}:${ws}"`
   - `resolve()` returns cached result immediately on second call for same key, skipping all `_fs.exists` calls
   - Public `clearCache(): void` method resets the cache (used in tests between calls)
   - Unit tests cover: second resolve returns cached value, `clearCache()` allows fresh resolution

## Dev Notes

### Background — Why This Story Exists

Story 2.5 (`ProjectService`) implemented email resolution as a private `resolveEmailLocation()` method, with this tech debt note:

> "Story 2.6 will formalize this as `EmailResolutionService`. When implemented, replace calls to this method with `EmailResolutionService.resolve()`."

This story extracts that logic into a standalone, reusable, cached service.

### New File: `src/types/email-resolution.types.ts`

```typescript
export interface IEntityLocation {
  /** Where the profile lives in the workspace */
  type: 'team' | 'leadership' | 'relationship';
  /** Absolute filesystem path to the profile .md file */
  absolutePath: string;
  /** True when auto-created during this resolution (step 4 of hierarchy) */
  created: boolean;
}
```

> **Naming note:** The epic uses the name `EntityLocation` with field `path`. This story uses `IEntityLocation` (following the `I` prefix convention) and `absolutePath` (to avoid ambiguity with relative paths used in wiki-links). The semantic meaning is identical.

### New File: `src/services/email-resolution.service.ts`

Full class shape:

```typescript
import path from 'node:path';
import { FileSystemService, fileSystemService } from './file-system.service.js';
import { RelationshipService, relationshipService } from './relationship.service.js';
import { getWorkspaceRoot as resolveWorkspaceRoot } from '../utils/workspace.js';
import type { IEntityLocation } from '../types/email-resolution.types.js';

export class EmailResolutionService {
  private readonly _cache: Map<string, IEntityLocation> = new Map();

  constructor(
    private readonly _fs: FileSystemService,
    private readonly _relationship: RelationshipService,
  ) {}

  getWorkspaceRoot(): string {
    return resolveWorkspaceRoot();
  }

  validateEmail(email: string): boolean {
    if (!email || typeof email !== 'string') return false;
    return /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(email.trim());
  }

  async resolve(email: string, ws: string): Promise<IEntityLocation> {
    const e = email.toLowerCase().trim();
    if (!this.validateEmail(e)) {
      throw new Error(`Invalid email address: '${email}'`);
    }
    const key = `${e}:${ws}`;
    const cached = this._cache.get(key);
    if (cached !== undefined) return cached;
    const result = await this._doResolve(e, ws);
    this._cache.set(key, result);
    return result;
  }

  generateWikiLink(email: string, resolvedPath: string, fromPath: string): string {
    const rel = path.relative(path.dirname(fromPath), resolvedPath);
    const normalizedRel = rel.split(path.sep).join('/');
    return `[[${normalizedRel}|${email.toLowerCase()}]]`;
  }

  clearCache(): void {
    this._cache.clear();
  }

  private async _doResolve(email: string, ws: string): Promise<IEntityLocation> {
    const teamProfile = path.join(ws, 'my-teams', '_members', email, `${email}.md`);
    if (await this._fs.exists(teamProfile)) {
      return { type: 'team', absolutePath: teamProfile, created: false };
    }

    const leaderProfile = path.join(ws, 'my-leadership', email, `${email}.md`);
    if (await this._fs.exists(leaderProfile)) {
      return { type: 'leadership', absolutePath: leaderProfile, created: false };
    }

    const relProfile = path.join(ws, 'my-company', 'relationships', email, `${email}.md`);
    if (await this._fs.exists(relProfile)) {
      return { type: 'relationship', absolutePath: relProfile, created: false };
    }

    // Step 4: auto-create as relationship
    await this._relationship.addRelationship(email, {}, ws);
    return { type: 'relationship', absolutePath: relProfile, created: true };
  }
}

export const emailResolutionService = new EmailResolutionService(
  fileSystemService,
  relationshipService,
);
```

### `generateWikiLink` Output Format

`generateWikiLink` returns `[[rel-path|email]]` WITHOUT the `- ` prefix. Callers prepend `- ` when building the wiki-link line for `appendToHashSection`:

```typescript

```

This matches the existing format: `- [[../../my-teams/_members/user@example.com/user@example.com.md|user@example.com]]`
