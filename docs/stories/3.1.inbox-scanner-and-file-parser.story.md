## Story

**As a** developer,
**I want** to scan and parse inbox files efficiently,
**so that** the process command can handle them.

## Acceptance Criteria

1. `InboxService` class with methods: `scanInbox()`, `parseFile()`
2. Supports file types: `.txt`, `.md`, `.json`
3. Returns array of `InboxFile` objects with: filepath, content, timestamp
4. Handles edge cases: empty inbox, large files, malformed files
5. Sorted by timestamp (oldest first)
6. Unit tests cover various file types and scenarios

interface InboxFile {

### FileSystemService API Available

All file I/O MUST go through `FileSystemService` (constructor-injected). Its relevant methods:

```typescript
listFiles(dirPath: string, extension?: string): Promise<string[]>
// Returns absolute paths of top-level files, optionally filtered by extension.
// Throws FileSystemError if directory doesn't exist.

readFile(filePath: string): Promise<string>
// Returns UTF-8 content. Throws FileSystemError on I/O error.
```

For `mtime`, use raw `fs-extra` stat — the service doesn't expose a `stat` method.
Import pattern: `import * as fsNs from 'fs-extra'` with the CJS/ESM interop shim already established in `file-system.service.ts`.

[Source: src/services/file-system.service.ts]

### Result<T> Wrapper

ALL public async methods that can fail MUST return `Result<T>` — never throw from async methods:

```typescript
type Result<T> =
  | { success: true; data: T }
  | { success: false; error: string };
```

[Source: docs/architecture/coding-standards.md#critical-rules]

### Supported File Types (AC: 2)

```typescript
const SUPPORTED_EXTENSIONS = ['.txt', '.md', '.json'] as const;
```

Scan strategy: call `listFiles(inboxDir, ext)` for each extension, then merge + sort.

### Large File Guard (AC: 4)

```typescript
const MAX_FILE_SIZE_BYTES = 10 * 1024 * 1024; // 10 MB
```

Check file size via `fs.stat(filePath).size` before reading. If `size > MAX_FILE_SIZE_BYTES`, return `{ success: false, error: \`File too large: ${filePath} (${size} bytes)\` }`.

### Error Handling for Edge Cases (AC: 4)

- **Empty inbox**: `listFiles` returns `[]` for each extension → `scanInbox` returns `{ success: true, data: [] }`. Do NOT treat this as an error.
- **Malformed files**: A `.json` file with invalid JSON is still "parseable" as raw text for this story — `parseFile` returns the raw string content. The categorization service (Story 3.2) is responsible for semantic parsing. So `parseFile` should NOT attempt JSON.parse.
- **Large files**: Guard in `parseFile`, return error result.
- **Unreadable files (fs errors)**: Catch `FileSystemError` from `readFile`, return `{ success: false, error: err.message }`.
- **`stat` errors**: Catch and return `{ success: false, error: ... }`.

### Security Review

No security concerns. No PII handling, no API calls, no credential exposure. The 10 MB large-file guard prevents memory exhaustion from malicious or oversized inbox files. File path is caller-provided (workspace root); path traversal is an architectural concern for the CLI layer, not this service.

### Performance Considerations

Sequential extension scanning is appropriate for MVP. `parseFile` reads content only after a size check, avoiding unnecessary I/O for oversized files. For Story 3.6's `tmr process --dry-run`, `parseFile` could be called with a size-only fast-path (future optimization).
