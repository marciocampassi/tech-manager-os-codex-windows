# Story 1.7: API Key Validation — Retry Limit & Deferred Setup

## Status

Done

## Story

**As a** new user running `tmr init`,
**I want** the API key validation to stop looping after 3 failed attempts and let me skip for now,
**so that** a bad key does not block the entire onboarding and I can finish setup and configure the key later.

## Acceptance Criteria

1. During `tmr init`, API key validation retries at most **3 times** total (first attempt + 2 retries).
2. Each failed attempt displays the clean error message returned by the provider (e.g. `API key not valid. Please pass a valid API key.`) and indicates the attempt number (e.g. `Attempt 1 of 3`).
3. After the 3rd failure, the wizard displays a warning message:
   > ⚠ Could not validate the API key after 3 attempts. You can configure it later with `tmr config set-key`.
4. After the warning, the wizard **continues** the remaining onboarding steps (profile, team members, workspace creation) without exiting.
5. When the key is skipped/deferred, the provider is still recorded in config (provider name only, no key), so the workspace is correctly created.
6. If the user presses Ctrl+C at any point, the program exits normally (no change to existing behaviour).
7. Unit tests cover:
   - 1st attempt valid → continues immediately.
   - 1st and 2nd fail, 3rd valid → continues.
   - All 3 fail → warning shown, onboarding continues without a key.
8. Integration test validates the 3-failure path end-to-end.

### Current Behaviour (from Story 1.6)

- `validateConnection()` returns `Promise<boolean>` (true = success, false = failure).
- `run()` has an unbounded `while (true)` loop: on failure it calls `promptApiKey` again indefinitely.
- Target: cap that loop at `MAX_API_KEY_ATTEMPTS = 3`.

### Deferred Key Handling

- If all 3 attempts fail, skip calling `addProvider` with the key.
- Still call `configService.setActiveProvider(provider)` so the workspace scaffolding knows which provider is intended.
- The key can be stored later via `tmr config set-key` (Story 1.8).

### Testing Standards

- Test files: `tests/commands/init.command.test.ts`, `tests/integration/init.integration.test.ts`
- Framework: Jest with `jest.unstable_mockModule` for ESM mocks
- Use `mockResolvedValueOnce` chain to simulate sequential prompt responses
- For the 3-failure scenario: mock `validateConnection` to return `false` three times, then assert warning message appears and subsequent prompts (profile, team members) are still called
