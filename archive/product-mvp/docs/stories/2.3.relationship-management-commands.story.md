# Story 2.3: Relationship Management Commands

## Status

Done

## Story

**As a** manager,
**I want** CLI commands to track company relationships outside my direct reports,
**so that** I maintain context on cross-functional collaborators and stakeholders.

## Acceptance Criteria

1. **`tmr relationship add <email-or-list>`** creates `my-company/relationships/{email}/{email}.md` and `my-company/relationships/{email}/1on1s/` for each email. If the relationship already exists, prints an "already exists" message without error. If no email provided, prompts interactively for: email, name, role, department, relationship_type.

2. **Batch mode**: when `<email-or-list>` contains commas, processes each email in the list sequentially, shows a progress indicator per email, and prints a summary: `X created, Y already existed`.

3. **`tmr relationship 1on1 <email> [--date=<date>] [--no-edit]`** creates `my-company/relationships/{email}/1on1s/{date}-{email}-1on1.md` using the relationship 1on1 template, and appends a wiki-link to the `## 1on1s` section of `{email}.md`. Error if relationship not found — suggests running `tmr relationship add <email>` first.

4. **`tmr relationship list`** displays a table of all relationships with columns: email, name, department, relationship_type, last 1on1 — sorted by most recent interaction first (relationships with no 1on1 appear at the bottom).

5. All `add` sub-operations: email normalized to lowercase; profile frontmatter fields `name`, `role`, `department`, `relationship_type` are optional strings (empty string if not supplied via flags or prompt).

6. **`TemplateService`** extended with `getRelationship1on1Template(date, email)` returning a 1on1 template specific to relationships (sections: Alignment Topics, Support Needed, Feedback Requested, Notes).

7. All commands have **unit tests** covering: creation, already-exists, interactive mode, batch, not-found error, listing. An **integration test** validates: create relationship → add 1on1 → verify file exists + wiki-link appended + list output correct.

## Dev Notes

### Directory Structure

```
my-company/
  relationships/
    {email}/
      {email}.md          ← profile file
      1on1s/              ← 1on1 files directory
```

### Profile Frontmatter (`{email}.md`)

```yaml
---
email: user@example.com
name: ''
role: ''
department: ''
relationship_type: ''
date_added: 2026-03-07
---

# Relationship — user@example.com

## Notes

## 1on1s
```

### 1on1 Template (Relationship — different from team member 1on1)

```markdown
---
date: 2026-03-07
member: user@example.com
type: 1on1
---

# 1:1 with user@example.com

## Alignment Topics

## Support Needed

## Feedback Requested

## Notes
```

### File Naming & Wiki-Link

- 1on1 file: `{date}-{email}-1on1.md` (same pattern as member)
- Wiki-link appended to profile: `- [[1on1s/{date}-{email}-1on1.md]]`
- Relative path works because `{email}.md` and `1on1s/` are siblings

### CLI Command Structure

```
tmr relationship add [email-or-list] [--name X] [--role X] [--department X] [--type X]
tmr relationship 1on1 <email> [--date YYYY-MM-DD] [--no-edit]
tmr relationship list
```

> **Note:** PRD specifies `tmr relationship <email> add 1on1` — Commander.js cannot support positional args + subcommands cleanly at the same level. Implemented as `tmr relationship 1on1 <email>` (consistent with the pattern established in Story 2.2 for member files). Documented deviation.

### `listRelationships` — Last Interaction Logic

`listFiles(1on1sDir)` returns **absolute paths** sorted alphabetically. Since files are named `{date}-{email}-1on1.md`, alphabetical = chronological. Last element = most recent.

```typescript
// Extract date from absolute path basename:
const baseName = path.basename(lastFilePath);           // '2026-03-07-user@co.com-1on1.md'
const dateMatch = baseName.match(/^(\d{4}-\d{2}-\d{2})/);
const lastInteraction = dateMatch ? dateMatch[1] : '-';
```

Sort order: most recent first, relationships with no 1on1 (`'-'`) at the bottom.
