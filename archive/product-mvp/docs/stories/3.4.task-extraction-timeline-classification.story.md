### Task File Format

Task files live at `{workspaceRoot}/my-tasks/{period}.md`. The format for each appended task is:

```markdown
- [ ] **{description}** — {owner}
  - Urgency: {urgencyReason}
  - Source: {sourceFile}
```

Example:
```markdown
- [ ] **Review John's PR before deployment** — john.doe@company.com
  - Urgency: Explicitly marked ASAP in meeting
  - Source: inbox/1on1-with-john-2026-04-07.md
```

When marking a task complete, the `- [ ]` prefix becomes `- [x]`:
```markdown
- [x] **Review John's PR before deployment** — john.doe@company.com
  - Urgency: Explicitly marked ASAP in meeting
  - Source: inbox/1on1-with-john-2026-04-07.md
```

### AI Prompt Design

**System Prompt (`TASK_EXTRACTION_SYSTEM_PROMPT`):**
```
You are a task extraction assistant for an engineering manager's workspace.
Analyze the provided meeting notes and inbox files. Your job is to:
1. Extract all actionable tasks and classify them by urgency
2. Identify any tasks from the existing task list mentioned as completed

Respond with ONLY a valid JSON object — no markdown, no explanation, no code fences.
The JSON must have this exact structure:
{
  "tasks": [
    {
      "description": "clear actionable task description",
      "owner": "person name or email responsible",
      "urgencyReason": "why this urgency level",
      "period": "today|this-week|this-month|this-quarter",
      "status": "todo"
    }
  ],
  "completedDescriptions": ["partial text matching an existing task that was mentioned as done"]
}

Classification rules:
- "today": Urgent items, "ASAP", "end of day", explicit today deadlines, overdue
- "this-week": "this week", "by Friday", "soon", week-level deadlines
- "this-month": Monthly goals, "end of month", month-level deadlines
- "this-quarter": Quarterly objectives, "this quarter", longer-horizon goals

Rules:
- "tasks" must be an array (empty if no tasks found)
- "completedDescriptions" must be an array (empty if no completions mentioned)
- "status" must always be "todo" for new tasks
- "period" must be one of the exact values listed above
- "owner" should be the email address if known, or the person's name if only name is available
- "description" should be concise and actionable (imperative mood: "Review PR", "Send update")
- Only extract tasks that require the manager or a team member to take an action
- Do NOT extract tasks that are already noted as complete in the current notes
```

### `appendFile` Method for `FileSystemService`

Add to `src/services/file-system.service.ts`:

```typescript
async appendFile(filePath: string, content: string): Promise<void> {
  let existing = '';
  try {
    existing = await this.readFile(filePath);
  } catch {
    // File does not exist — will be created
  }
  await this.writeFile(filePath, existing + content);
}
```

**Rationale:** Uses the existing atomic `writeFile()` method internally (tmp → rename). `readFile()` failure (FileSystemError) silently means "create new file" — no explicit exists check needed.

### Marking Completed Tasks

`_markCompletedInContent(content: string, completedDescriptions: string[])`:

For each description in `completedDescriptions`, perform a case-insensitive partial match:
- Search for lines matching: `- [ ] **...{description}...` (the description may be a substring of the full task title)
- Replace `- [ ]` with `- [x]` on that line only
- Count replacements

Implementation approach (no regex needed):
```typescript
private _markCompletedInContent(
  content: string,
  completedDescriptions: string[],
): { updated: string; markedCount: number } {
  let result = content;
  let markedCount = 0;
  for (const desc of completedDescriptions) {
    const lower = desc.toLowerCase();
    const lines = result.split('\n');
    let replaced = false;
    const updated = lines.map((line) => {
      if (!replaced && line.startsWith('- [ ] **') && line.toLowerCase().includes(lower)) {
        replaced = true;
        markedCount++;
        return line.replace('- [ ] **', '- [x] **');
      }
      return line;
    });
    result = updated.join('\n');
  }
  return { updated: result, markedCount };
}
```

### AI Response Runtime Validation

Validate the AI JSON payload with a type-guard (no Zod — not installed):

```typescript
function isExtractedTaskArray(v: unknown): v is Array<{
  description: string; owner: string; urgencyReason: string;
  period: TaskPeriod; status: string; sourceFile?: string;
}> {
  if (!Array.isArray(v)) return false;
  return v.every(
    (t) =>
      typeof t === 'object' && t !== null &&
      typeof (t as Record<string, unknown>)['description'] === 'string' &&
      typeof (t as Record<string, unknown>)['owner'] === 'string' &&
      typeof (t as Record<string, unknown>)['urgencyReason'] === 'string' &&
      TASK_PERIODS.has((t as Record<string, unknown>)['period'] as TaskPeriod) &&
      (t as Record<string, unknown>)['status'] === 'todo'
  );
}
```

Export `const TASK_PERIODS: ReadonlySet<TaskPeriod>` from `task.types.ts` for use in the type-guard (matches the `NOTE_TYPES` pattern from `categorization.types.ts`).

The AI does NOT return `sourceFile` — the service injects `file.filepath` into each task after extraction (since the AI processes multiple files at once, attribute each task to the correct source using the prompt structure or let the AI include it).

**Simplification:** Have the AI include `sourceFile` in its JSON by injecting the file paths into the system prompt. Or: since tasks are extracted per-file in a batch, the `sourceFile` can be left as `''` in the AI JSON and filled in by the service based on which file the task was mentioned in. For simplicity: let the AI fill `sourceFile` from the content, and if absent default to the first file's path.

**IMPORTANT:** The `status` field returned by the AI is always `'todo'` — validation rejects anything else. The `sourceFile` field in `ExtractedTask` is optional in the AI response (type-guard allows absent/empty) and defaults to `files[0]?.filepath ?? ''` when missing.

### Error Handling

- `_ai.generateText()` throws → catch, return `{ success: false, error: err.message }`
- `_parseAIResponse()` returns `{ success: false }` → propagate immediately
- `_loadExistingTasks()` — per-period errors are silenced (return `''`); never fail the whole extraction
- `_fs.appendFile()` throws → catch, return `{ success: false, error: err.message }`
- `_fs.writeFile()` (for completion marking) throws → catch, return `{ success: false, error: err.message }`
- Never throw from any public method — always return `Result<TaskExtractionResult>`

### Code Quality Assessment

Excellent implementation. `TaskService` follows all project conventions: `Result<T>` wrapper (inline), constructor DI, `_` prefix private members, no singleton export, explicit return types, no `any`. Notable design choices worth calling out:

Non-atomic appends are acceptable for task files since concurrent writers aren't expected in the single-user CLI architecture.
- **`_markCompletedInContent` uses `split('\n')` + `map`** rather than regex — cleaner, avoids escaping complexity, correctly handles the first-match-per-description constraint. ✅
- **`TASK_PERIODS` and `TASK_FILE_MAP` exported from types** — makes them reusable for `TaskViewService` unification in Story 3.6 without introducing a shared util module. ✅
