# Story 2.5: Project Management Commands

## Status

Done

## Story

**As a** manager,
**I want** CLI commands to manage project structures and team composition,
**so that** I track project context and stakeholder relationships.

## Acceptance Criteria

1. **`tmr project add [project-name]`** creates two structures: `/my-company/projects/{project-name}-project.md` (project overview with frontmatter) and `/my-projects/{project-name}/` directory containing `{project-name}-composition.md`, `standup/`, `discussion/`, and `presentation/` subdirectories. If no project-name provided, prompts interactively. If project already exists, prints "already exists" without error.

2. **`tmr project standup <project-name> [--date=<date>] [--no-edit]`** creates `/my-projects/{project-name}/standup/{date}-{project-name}-standup.md` using the standup template (sections: Yesterday, Today, Blockers). Errors if project not found.

3. **`tmr project discussion <project-name> [--date=<date>] [--no-edit]`** creates `/my-projects/{project-name}/discussion/{date}-{project-name}-discussion.md` using the discussion template (sections: Topic, Attendees, Decisions, Action Items). Errors if project not found.

4. **`tmr project presentation <project-name> --topic <topic> [--date=<date>] [--no-edit]`** creates `/my-projects/{project-name}/presentation/{date}-{project-name}-presentation-{topic-slug}.md` using the presentation template (sections: Slides Outline, Talking Points, Q&A). Prompts interactively for topic if not supplied. Errors if project not found.

5. **`tmr project link-member <project-name> <email>`** resolves the email using inline hierarchical resolution (team → leadership → relationship; creates as relationship if not found anywhere), then appends `- [[{relative-path}|{email}]]` to the `# Team Members` section of `{project-name}-composition.md`. Returns `{ linked, created }` where `created` is true if a new relationship was auto-created.

6. **`tmr project link-members <project-name> <email-list>`** accepts a comma-separated list and executes `link-member` for each; shows progress per email and prints summary `X linked, Y created`.

7. **`tmr project link-stakeholder <project-name> <email>`** same logic as `link-member` but appends to `# Stakeholders` section.

8. **`tmr project link-stakeholders <project-name> <email-list>`** batch version of `link-stakeholder`.

9. **`tmr project list`** displays a table: project-name, member count, stakeholder count — by parsing `{project-name}-composition.md` and counting `- [[` lines in each section.

10. **`TemplateService`** extended with five new methods: `getProjectOverviewTemplate(name, date)`, `getProjectCompositionTemplate()`, `getStandupTemplate(date, name)`, `getDiscussionTemplate(date, name)`, `getPresentationTemplate(date, name, topic)`.

11. All commands have **unit tests** covering: creation, already-exists, not-found, interactive, batch, listing. An **integration test** validates the full workflow: create → add standup → link-member → link-stakeholder → list shows correct counts.

### Composition Template (`{project-name}-composition.md`)

```markdown

```

> **Note:** Composition uses `# ` (single hash) headers, NOT `## ` (double hash). This is intentional — it separates the two top-level groupings. `SectionParserService` uses `## ` and is NOT used here. `ProjectService` has a private `appendToHashSection()` that handles `# ` sections. See Dev Notes → Private Helpers.

### CLI Command Structure (Commander.js)

```
tmr project add [project-name] [--date X]
tmr project standup <project-name> [--date YYYY-MM-DD] [--no-edit]
tmr project discussion <project-name> [--date YYYY-MM-DD] [--no-edit]
tmr project presentation <project-name> [--topic X] [--date YYYY-MM-DD] [--no-edit]
tmr project link-member <project-name> <email>
tmr project link-members <project-name> <email-list>
tmr project link-stakeholder <project-name> <email>
tmr project link-stakeholders <project-name> <email-list>
tmr project list
```

> **PRD Deviation (Commander.js constraint):** The PRD specifies `tmr project <project-name> add standup`. Commander.js cannot cleanly support a positional arg before a subcommand at the same level. The same deviation was applied in Stories 2.3 (relationship) and 2.4 (leadership). Subcommand comes first: `tmr project standup <project-name>`.

### Private Helpers in `ProjectService`

**`appendToHashSection(content, sectionName, line)`**

Handles `# ` section append (NOT `## `). Mirrors `SectionParserService.appendToSection` but looks for `# {sectionName}`:

```typescript
private appendToHashSection(content: string, sectionName: string, line: string): string {
  const lines = content.split('\n');
  const secIdx = lines.findIndex((l) => l === `# ${sectionName}`);
  if (secIdx === -1) {
    return content.trimEnd() + `\n\n# ${sectionName}\n\n${line}\n`;
  }
  let nextSecIdx = lines.findIndex((l, i) => i > secIdx && l.startsWith('# '));
  if (nextSecIdx === -1) nextSecIdx = lines.length;
  const before = lines.slice(0, nextSecIdx).join('\n').trimEnd();
  const after = lines.slice(nextSecIdx).join('\n');
  return after.length > 0 ? `${before}\n${line}\n\n${after}` : `${before}\n${line}\n`;
}
```

**`countHashSection(content, sectionName)`**

Counts `- [[` lines in a `# ` section:

```typescript
private countHashSection(content: string, sectionName: string): number {
  const lines = content.split('\n');
  const secIdx = lines.findIndex((l) => l === `# ${sectionName}`);
  if (secIdx === -1) return 0;
  let nextSecIdx = lines.findIndex((l, i) => i > secIdx && l.startsWith('# '));
  if (nextSecIdx === -1) nextSecIdx = lines.length;
  return lines.slice(secIdx + 1, nextSecIdx).filter((l) => l.startsWith('- [[')).length;
}
```

### Inline Email Resolution (`resolveEmailLocation`)

Story 2.6 will formalize this as `EmailResolutionService`. For now, `ProjectService` holds it as a private method. `RelationshipService` is injected into `ProjectService` for the auto-create step (step 4).

```
Hierarchy:
1. /my-teams/_members/{email}/{email}.md    → type: 'team'
2. /my-leadership/{email}/{email}.md        → type: 'leadership'
3. /my-company/relationships/{email}/...    → type: 'relationship'
4. Not found → call this._relationship.addRelationship(email, {}, ws) → type: 'relationship'
```

Relative path from `my-projects/{project-name}/` (2 dirs deep from workspace root):
- Team:         `../../my-teams/_members/{email}/{email}.md`
- Leadership:   `../../my-leadership/{email}/{email}.md`
- Relationship: `../../my-company/relationships/{email}/{email}.md`

Wiki-link format: `- [[../../my-teams/_members/{email}/{email}.md|{email}]]`

**Tech Debt Note:** This logic duplicates what Story 2.6 will formalize as `EmailResolutionService`. When Story 2.6 is implemented, `resolveEmailLocation` in `ProjectService` should be removed and replaced with a call to `EmailResolutionService.resolve()`.

> `SectionParserService` is NOT injected — `ProjectService` uses its own `appendToHashSection` for `# ` sections. The composition file format (`# ` headers) is incompatible with `SectionParserService` (`## ` headers).
