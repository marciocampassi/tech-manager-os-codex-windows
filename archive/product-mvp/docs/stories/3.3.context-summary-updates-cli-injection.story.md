# Story 3.3: Context Summary Updates via CLI Injection

## Status

Ready for Done

## Story

**As a** manager,
**I want** context summaries intelligently updated using CLI commands,
**so that** I have current, accurate context without token-heavy file reading operations.

## Acceptance Criteria

1. `ContextService` class with method: `updateContext(email, insights)`
2. Instead of reading full context files, executes CLI commands:
   - For team members: Uses Epic 2 CLI to append insights
   - For projects: Updates project context via CLI
   - For relationships/leadership: Updates via CLI
3. AI generates insights from new notes
4. CLI handles section parsing and atomic updates
5. Token optimization: AI only generates insights, doesn't manipulate files
6. Unit tests with mocked CLI execution
7. Integration tests validate context updates

## Dev Notes

### Epic Context

Story 3.3 is the third story of Epic 3 (Process Intelligence Engine). It implements the **token-optimization layer** for context updates:

- **Depends on:** Story 3.1 (`InboxService`, `InboxFile` — complete) and Story 3.2 (`CategorizationResult.insights`, `CategorizationResult.members`, `CategorizationResult.projects` — complete)
- **Consumed by:** Story 3.6 (`tmr process` command) — the orchestrator calls `updateContext()` / `updateProjectContext()` for each entity in the categorization result

**The CLI Injection Pattern:** Instead of sending full context files to the AI for updating (token-expensive), the AI generates compact insights during categorization (Story 3.2). This service then "injects" those insights into the right context files using the Epic 2 service layer (`SectionParserService`). The AI never reads or writes context files directly.

### Data Models

**`EntityType`** (string union):
```typescript
type EntityType = 'member' | 'project' | 'leadership';
```

**`ContextUpdateResult`** (all fields required):
```typescript
interface ContextUpdateResult {
  entityType: EntityType;      // which entity type was updated
  identifier: string;          // email address or project name slug
  contextFilePath: string;     // absolute path to the context.md file written
  insightsAppended: number;    // number of insight strings written (may be 0)
  created: boolean;            // true when context.md did not exist and was created
}
```

**Insight Block Format** appended to `## Context Log` section:
```markdown
### 2026-04-07
- Insight text 1
- Insight text 2
```

Each call to `_appendInsights` appends ONE dated block (using today's ISO date). If insights array is empty, no block is written and `insightsAppended` is 0.

[Source: docs/prd/epics/epic-3-process-intelligence-engine.md#story-33]

### Workspace Path Structure

The `ContextService` constructs context.md paths using these path conventions (derived from existing service code):

| Entity | Directory | Context File Path |
|--------|-----------|-------------------|
| Member | `my-teams/members/{email}/` | `my-teams/members/{email}/context.md` |
| Project | `my-company/projects/{name}-project/` | `my-company/projects/{name}-project/context.md` |
| Leadership | `my-leadership/{email}/` | `my-leadership/{email}/context.md` |

These match the path helpers in `src/services/member.service.ts`, `project.service.ts`, and `leadership.service.ts` respectively.

**IMPORTANT:** The project name passed to `updateProjectContext()` is the raw name (e.g., `'api-redesign'`). The service must normalize it by appending `-project` suffix if not already present (same logic as `normalizeProjectName()` in `project.service.ts`):

```typescript
function normalizeProjectName(name: string): string {
  return name.endsWith('-project') ? name : `${name}-project`;
}
```

[Source: src/services/member.service.ts, src/services/project.service.ts, src/services/leadership.service.ts]

### Entity Type Detection Logic (updateContext)

`updateContext(email, insights, workspaceRoot)` must auto-detect entity type:

1. Check `{ws}/my-teams/members/{email.toLowerCase()}` directory exists → entity is `'member'`
2. Else check `{ws}/my-leadership/{email.toLowerCase()}` directory exists → entity is `'leadership'`
3. Else return `{ success: false, error: 'Entity not found for email: {email}' }`

Normalize `email` to lowercase before any path construction (consistent with all Epic 2 services).

### Context File Creation

When `context.md` does not exist for an entity, create it with a `## Context Log` header before appending:

```typescript
const initialContent = '# Context Log\n\n## Context Log\n';
await this._fs.writeFile(contextFilePath, initialContent);
```

Then call `_sectionParser.appendToFile()` normally. The `created: true` flag in the result signals the caller (Story 3.6) that a new file was bootstrapped.

**Do NOT create the parent directory** — assume the parent directory exists (it was created when the entity was added via `tmr team add` / `tmr project add` / `tmr leadership add`). If the parent does not exist, `_fs.writeFile()` will throw, which is caught and returned as `{ success: false, error }`.

### AI Integration (for reference — not implemented here)

Story 3.3 itself does NOT call the AI. The AI generates insights during Story 3.2's categorization step. Story 3.3 only consumes the already-generated `insights: Record<string, string[]>` from `CategorizationResult`.

The calling flow (Story 3.6):
1. `CategorizationResult.insights` → `{ "John Doe": ["career goal discussed"], "api-redesign": ["on track"] }`
2. Resolve `"John Doe"` → email `john.doe@company.com` (via email resolution service)
3. Call `contextService.updateContext("john.doe@company.com", ["career goal discussed"], ws)`
4. Call `contextService.updateProjectContext("api-redesign", ["on track"], ws)`

[Source: docs/prd/epics/epic-3-process-intelligence-engine.md#story-33]

### Error Handling

- `_fs.exists()` throws → catch, return `{ success: false, error: err.message }`
- `_fs.writeFile()` throws (creating context.md) → catch, return `{ success: false, error: err.message }`
- `_sectionParser.appendToFile()` throws → catch, return `{ success: false, error: err.message }`
- Never throw from any public method — always return `Result<ContextUpdateResult>`

### Refactoring Performed

- **File:** `src/services/context.service.ts`
  - **Change:** Added leading `\n` to the insight block returned by `_formatInsightBlock()` — changed `\`### ${date}\n${items}\`` to `\`\n### ${date}\n${items}\``
  - **Why:** Without the leading newline, consecutive calls append dated blocks with no visual separation: `### 2026-04-07\n- A\n- B\n### 2026-04-08\n- C`. With it, `SectionParserService.trimEnd()` + the `\n` prefix produce a clean blank line between entries: `### 2026-04-07\n- A\n- B\n\n### 2026-04-08\n- C`.
  - **How:** Single character addition to the template literal in `_formatInsightBlock`. No API change; pure formatting improvement.

- **File:** `tests/services/context.service.test.ts`
  - **Change:** Updated the ISO date regex from `^### \d{4}-\d{2}-\d{2}` to `### \d{4}-\d{2}-\d{2}` (removed `^` anchor, added a new test specifically for the leading newline)
  - **Why:** The original `^` anchor test would have failed after the block now starts with `\n`. Separating "has date" from "has leading newline" improves test intent clarity.
  - **How:** One test updated, one test added: `'should prefix the insight block with a leading newline for block separation'`

- **File:** `tests/services/context.service.integration.test.ts`
  - **Change:** Added assertion to the "append second call" test that verifies a blank line exists between the two dated blocks
  - **Why:** Regression coverage for the block-separator refactoring above
  - **How:** Slices the content between the two insight sets and asserts `\n\n` is present
