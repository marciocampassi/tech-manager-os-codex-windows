# Story 2.1: Team Management Commands

## Status

Done

## Story

**As a** manager,
**I want** CLI commands to manage teams and team members,
**so that** I can organize my direct reports across multiple teams with a single source of truth per person.

## Acceptance Criteria

1. **Directory structure** matches the PRD `_members/` / `_teams/` layout (detailed in Dev Notes).
2. **`tmr team create <team-name>`** creates team directory structure with context and members files; interactive fallback if no name given.
3. **`tmr team add <team-name> <email> [--role=<role>] [--location=<location>]`** creates member directory and profile with correct frontmatter and sections; auto-creates the team if it doesn't exist; if member already exists in another team, appends team to their `teams` array; appends wiki-link to team's members file; interactive fallback if no parameters given.
4. **`tmr team list [team-name]`** without argument shows table of all teams with member counts; with `team-name` shows table of that team's members (email, role, location, date_added).
5. **`tmr team archive <team-name> <email> [--from <date>] [--to <date>]`** moves member directory to `_archived/{year}/{email}/`, adds `archived: true` and `archived_date` to frontmatter, removes wiki-link from team members file; interactive fallback; supports optional date-range filter.
6. **`tmr team fire <team-name> <email>`** executes archive logic and additionally adds `termination: true` and `termination_date` to frontmatter.
7. **`tmr show <email>`** searches across all locations in priority order and displays the profile; returns a clear "not found" message when missing.
8. All commands have **unit tests** covering the scenarios listed in Dev Notes.
9. An **integration test** validates the full create → add → list → archive workflow on a temp workspace.

## Dev Notes

### ⚠️ Architecture Discrepancy — Data Models vs PRD

`docs/architecture/data-models.md` defines `my-teams/{team}/{email}/` (team in path). **Epic 2 PRD overrides this** with a separate `_members/` and `_teams/` structure to support multi-team membership. Use the **PRD structure** for all implementation:

```
my-teams/
  _members/{email}/                  ← single canonical location per person
    {email}.md                       ← identity anchor + frontmatter + sections
    1on1s/
    feedback/
    assessments/
    performance-reviews/
  _teams/{team-name}/                ← team metadata
    {team-name}-context.md
    {team-name}-members.md           ← wiki-links to _members/
  _archived/{year}/{email}/          ← archived members
```

### Manager Email Resolution

The `addMember` method needs the manager's email for the "Current Manager" section. Read it from `my-career/profile.md` frontmatter using `gray-matter` (already in the tech stack):

```typescript
import matter from 'gray-matter';

async function getManagerEmail(workspaceRoot: string): Promise<string | null> {
  const profilePath = path.join(workspaceRoot, 'my-career', 'profile.md');
  if (!(await fileSystemService.exists(profilePath))) return null;
  const content = await fileSystemService.readFile(profilePath);
  const { data } = matter(content);
  return (data['email'] as string | undefined) ?? null;
}
```

```markdown

```

**Wiki-link line appended** per `tmr team add` call (path is relative from `_teams/{team}/` up to `_members/`):

```

```

### Archive Logic (`tmr team archive`)

1. Determine target: `my-teams/_archived/{year}/{email}/`
2. Use `fileSystemService.moveFile()` for the directory (fs-extra `move` handles dirs)
3. Update frontmatter in the moved `{email}.md`:
   - Add `archived: true`
   - Add `archived_date: <ISO date>`
4. Remove the wiki-link line for this member from `_teams/{team-name}/{team-name}-members.md`

For `--from`/`--to` date range filters: only move files (not the whole directory) whose filename starts with a date within the range. Move only those files; leave the directory in place.

For `tmr team fire`: call `archiveMember()` then additionally update frontmatter with `termination: true` and `termination_date: <ISO date>`.

### `tmr show <email>` Search Order

Search in this exact priority order, return the first match:
1. `my-teams/_members/{email}/` (active team member)
2. `my-teams/_archived/**/{email}/` (glob or iterate years)
3. `my-leadership/{email}/` (leadership)
4. `my-company/relationships/{email}/` (relationship)

Display the profile markdown content. If not found, print:
`Profile not found for <email>. Use 'tmr team add' or 'tmr relationship add' to create one.\n`

## QA Results

### Review Date: 2026-03-07

### Reviewed By: Quinn (Test Architect)

### Code Quality Assessment

Solid implementation overall. `TeamService` is well-structured with clean constructor injection enabling full testability. The Commander.js command pattern correctly follows project conventions. One non-trivial risk: the `archiveMember` date-range filter uses filesystem operations that were initially bypassing `FileSystemService`, and the date-filter code path had no integration test coverage.

### Refactoring Performed

- **File**: `src/services/file-system.service.ts`
  - **Change**: Added `listDirectories(dirPath)` method returning `string[]` of child directory names
  - **Why**: `TeamService` needed to enumerate team/archive directories but was using direct `import('node:fs/promises').readdir` calls, violating the project's "all file ops via FileSystemService" coding standard
  - **How**: `listDirectories` follows the same error-handling and typing pattern as `listFiles`, making it testable via mock

- **File**: `src/services/team.service.ts`
  - **Change**: Replaced 3 inline `await import('node:fs/promises')` + `readdir` calls with `this._fs.listDirectories()` in `listTeams()`, `showProfile()`, and the date-filter branch of `archiveMember()`
  - **Why**: Eliminate coding standards violation; FSS mock now fully covers all code paths in unit tests
  - **How**: All directory-listing operations now route through the injected `FileSystemService`

- **File**: `src/services/team.service.ts` — `archiveMember` date-filter branch
  - **Change**: Replaced `readdir({ recursive: true })` (Node 20.x feature, brittle) with explicit iteration over known subdirectory names via `FileSystemService.listFiles()`
  - **Why**: The recursive readdir approach had platform inconsistencies and bypassed FSS; the subdirectory names are a fixed schema (`1on1s`, `feedback`, `assessments`, `performance-reviews`)
  - **How**: Safer and fully testable without native fs calls

- **File**: `src/commands/team.command.ts`
  - **Change**: `runShow()` now uses `teamService` singleton instead of `new TeamService(fileSystemService)`
  - **Why**: Creates an inconsistent second instance with its own workspace root calculation; all commands should share the same singleton
  - **How**: Removed the redundant instantiation; also removed unused `fileSystemService` import

- **File**: `tests/services/team.service.test.ts`
  - **Change**: Added `listDirectories` to `MockFS` type and `createMockFS()` factory
  - **Why**: Mock was missing the new method, causing type errors and potential test gaps
