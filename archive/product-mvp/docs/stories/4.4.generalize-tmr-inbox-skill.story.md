### File Classification Logic

The skill classifies each inbox file into one of four categories:

| Category | Signal | Destination |
|----------|--------|-------------|
| **1:1** | Title contains a person's name/email, or is a meeting note between owner and one other person | `my-teams/members/{person-email}.md` |
| **Project** | References a named project, contains project keywords | `my-company/projects/{project-name}/` |
| **Company** | References company-wide topics, org changes, multiple people | `my-company/members/{email}.md` |
| **Ambiguous** | Cannot confidently classify | Left in inbox with `needs-review: true` tag added |

Domain inference: if a person appears without an email, infer `{first-name-lowercase}@{OWNER_DOMAIN}` as the default. For the vault owner themselves, use `OWNER_EMAIL` exactly.

### Profile Scaffolding

When a 1:1 or member profile does not exist yet:
- `my-teams/members/{email}.md` — create with frontmatter: `name`, `email`, `role`, `team`, `startDate`
- `my-company/members/{email}.md` — create with frontmatter: `name`, `email`, `department`
- `my-leadership/{slug}.md` — create only when a leadership framework or principle note is identified

### File Renaming Pattern

Archive destination: `archive/{category}/{YYYY-MM-DD}-{slug}.md`
- `YYYY-MM-DD` — from file frontmatter `date` field, or file modification date, or today
- `{slug}` — lowercase kebab-case of the file title

One functional correction was applied: the Dataview queries in the four view files used `contains(period, "today")` which assumes a `period` YAML field on tasks. Obsidian Dataview TASK queries do not automatically expose custom frontmatter on individual checkbox items. The correct pattern — `FROM "my-tasks/tasks.md" WHERE !completed AND contains(text, "(today)")` — matches the actual task format that the skill appends (`- [ ] description (period)`).

### Refactoring Performed

- **File**: `skills/tmr-inbox/SKILL.md`
  - **Change**: Replaced `WHERE !completed AND contains(period, "today/this-week/this-month/this-quarter")` with `FROM "my-tasks/tasks.md" WHERE !completed AND contains(text, "(today/this-week/this-month/this-quarter)")` in all four Dataview view file templates
  - **Why**: The original queries referenced `period` as if it were a Dataview-queryable YAML field on task items, which is not how Dataview TASK queries work. The skill's task extraction format embeds the period in the task text string as `(period)`, so `contains(text, ...)` is the correct filter
  - **How**: `FROM "my-tasks/tasks.md"` scopes the query to the tasks file, and `contains(text, "(today)")` matches the trailing period tag in each task line — functional and accurate

- **File**: `docs/stories/4.4.generalize-tmr-inbox-skill.story.md` (Dev Notes section only — no AC/Story sections modified)
  - **Change**: Updated the Dataview view file content examples in Dev Notes to match the corrected queries
