# Story 2.4: Leadership Management Commands

## Status

Done

## Story

**As a** manager,
**I want** CLI commands to track my leadership relationships,
**so that** I maintain context on my manager and skip-level leaders.

## Acceptance Criteria

1. **`tmr leadership add <email>`** creates the directory structure `/my-leadership/{email}/` containing `{email}.md` (profile with frontmatter) and `1on1s/` (directory). If the leadership already exists, prints an "already exists" message without error. If no email provided, prompts interactively for: email, name, role, areas_of_responsibility.

2. **`tmr leadership 1on1 <email> [--date=<date>] [--no-edit]`** creates `/my-leadership/{email}/1on1s/{date}-{email}-1on1.md` using the leadership 1on1 template (sections: Alignment Topics, Support Needed, Feedback Requested, Notes), and appends a wiki-link to the `## 1on1s` section of `{email}.md`. Error if leadership not found — suggests running `tmr leadership add <email>` first.

3. **`tmr leadership list`** displays a table of all leadership contacts with columns: email, name, role, last 1on1 — sorted by most recent interaction first (contacts with no 1on1s appear at the bottom).

4. All `add` operations: email normalized to lowercase; profile frontmatter fields `name`, `role`, `areas_of_responsibility` are optional strings (empty string if not supplied via flags or prompt).

5. **`TemplateService`** extended with `getLeadership1on1Template(date, email)` returning a leadership-specific 1on1 template (sections: Alignment Topics, Support Needed, Feedback Requested, Notes) with `type: leadership-1on1` in frontmatter to distinguish from the relationship variant.

6. All commands have **unit tests** covering: creation, already-exists, interactive mode, not-found error, listing. An **integration test** validates: create leadership → add 1on1 → verify file exists + wiki-link appended + list output correct.

## Dev Notes

### Directory Structure

```
my-leadership/
  {email}/
    {email}.md          ← profile file
    1on1s/              ← 1on1 files directory
```

### Profile Frontmatter (`{email}.md`)

```yaml
---
email: user@example.com
name: ''
role: ''
areas_of_responsibility: ''
date_added: 2026-03-09
---

# Leadership — user@example.com

## Notes

## 1on1s
```

Build the profile content using `matter.stringify()` (already used by `RelationshipService`) — import `gray-matter` as `matter`.

### 1on1 Template (Leadership-specific)

```markdown
---
date: 2026-03-09
member: user@example.com
type: leadership-1on1
---

# 1:1 with user@example.com

## Alignment Topics

## Support Needed

## Feedback Requested

## Notes
```

Note: `type: leadership-1on1` distinguishes from the relationship 1on1 (`type: 1on1`). Add `getLeadership1on1Template(date, email)` as a new public method on `TemplateService` — do NOT reuse `getRelationship1on1Template`.

### File Naming & Wiki-Link

- 1on1 file: `{date}-{email}-1on1.md` (same pattern as relationship)
- Wiki-link appended to profile: `- [[1on1s/{date}-{email}-1on1.md]]`
- Relative path works because `{email}.md` and `1on1s/` are siblings

### CLI Command Structure (Commander.js)

```
tmr leadership add [email] [--name X] [--role X] [--areas X]
tmr leadership 1on1 [email] [--date YYYY-MM-DD] [--no-edit]
tmr leadership list
```

> **Note:** Following the Commander.js constraint established in Story 2.3, the 1on1 subcommand is `tmr leadership 1on1 <email>` (NOT `tmr leadership <email> add 1on1` as stated in the epic). This pattern is consistent across member, relationship, and now leadership commands.

### `listLeadership` — Last Interaction Logic

Mirrors the exact pattern from `RelationshipService.listRelationships()`:

```typescript
// listFiles() returns absolute paths sorted alphabetically (= chronological by date prefix)
const files = await this._fs.listFiles(oneOnOneDir, '.md');
const lastFile = files[files.length - 1];
const baseName = path.basename(lastFile);
const dateMatch = baseName.match(/^(\d{4}-\d{2}-\d{2})/);
const lastInteraction = dateMatch ? dateMatch[1] : '-';
```

Sort: most recent first; contacts with no 1on1s (`'-'`) at the bottom.
