# Story 1.8: API Key Security Transparency & Key Management Commands

## Status

Done

## Story

**As a** user who has configured an API key with `tmr init`,
**I want** to inspect how my key is stored, delete it, update it, or switch providers from the CLI,
**so that** I can trust that my credentials are handled securely and retain full control over them.

## Acceptance Criteria

### Security transparency
1. `tmr config show-security` prints a human-readable security summary:
   - Storage path: `~/.config/tmr/config.json`
   - Encryption: AES-256 via the `conf` library (`encryptionKey: 'tmr-config-v1'`)
   - Stored value of the active key: redacted display (e.g. `AIza••••••••••••••••••••••••••••XYZ`)
   - A statement that the raw key is **never logged or printed anywhere**
2. The redacted display uses the existing `redact()` utility (`src/utils/redact.ts`).
3. `tmr config show-security` exits with code 0 and has no side effects.

### Key management
4. `tmr config set-key` starts an interactive prompt to:
   - Select provider (Gemini / OpenAI / Claude) — same `select` list as `tmr init`
   - Input the new API key (password-style, hidden)
   - Validate the key with a spinner (same `validateConnection` logic from `InitCommand`)
   - On success: persist and display `✔ Key saved for <provider>.`
   - On failure: display error message, re-prompt (up to 3 attempts — same cap as Story 1.7)
5. `tmr config delete-key` starts an interactive prompt to:
   - Select which provider key to delete (shows only providers that have a stored key)
   - Ask for confirmation: `Are you sure you want to delete the <provider> API key? [y/N]`
   - On confirmation: delete the key from config, display `✔ Key for <provider> deleted.`
   - On abort: display `Aborted.` and exit 0
6. `tmr config switch-provider` starts an interactive prompt to:
   - Show a `select` list of available providers (providers with a stored key are marked ✔)
   - Persist the selected provider as `active_provider`
   - Display `✔ Active provider set to <provider>.`
7. Running `tmr config` with no subcommand displays help listing the four subcommands.
8. All subcommands handle the case where no providers are configured yet (graceful message, exit 0).

### Engineering
9. A new `src/commands/config.command.ts` file implements the above subcommands using Commander.js sub-commands.
10. `config.command.ts` is registered in `src/cli.ts`.
11. Unit tests cover:
    - `show-security` output format and redacted key
    - `set-key` success path and 3-failure path
    - `delete-key` with confirmation and with abort
    - `switch-provider` persists correct value
12. No raw API key is ever passed to `console.log`, `logger`, spinners, or error messages.

### Storage facts (for `show-security` output)

From `config.service.ts`:
- **Path:** `~/.config/tmr/config.json` (via `Conf` with `cwd: CONFIG_DIR`)
- **Encryption:** `conf` uses AES-256 encryption when `encryptionKey` is provided (`'tmr-config-v1'`)
- **Key field name on disk:** `api_key_encrypted` inside each provider object
- **Redaction:** Use existing `configService.getRedacted('providers')` or call `redact()` directly on the raw key

### Story 1.7 dependency

Story 1.7 introduces the `MAX_API_KEY_ATTEMPTS = 3` cap on the init loop. `set-key` in this story must apply the same cap (3 attempts) to be consistent.

### Coding Standards

- Never use `console.log` — use `logger.info()` or `process.stdout.write` for display-only output
- No raw API key in any log or error message — always use `redact()`
- New command file follows kebab-case (`config.command.ts`)
- Constructor injection for `ConfigService` to support testability

### Testing Standards

- New test file: `tests/commands/config.command.test.ts`
- Framework: Jest with `jest.unstable_mockModule` for ESM mocks
- Mock `ConfigService` methods (`getProviderConfig`, `addProvider`, `delete`, `setActiveProvider`)
- Mock `inquirer` prompts with `mockResolvedValueOnce` for sequential responses
- Assert `process.stdout.write` / spinner calls for output verification
