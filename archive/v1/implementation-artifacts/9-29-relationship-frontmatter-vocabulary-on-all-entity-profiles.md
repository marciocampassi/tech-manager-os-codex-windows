# Story 9.29: Relationship Frontmatter Vocabulary on All Entity Profiles

Status: done

## Story

As a vault owner,
I want every entity profile (member, contractor, company-member, and team-member via `tmr team add`) to use a consistent relationship vocabulary in frontmatter,
so that managers, peers, projects, and history are all queryable as structured graph edges.

## Acceptance Criteria

**AC1 — Direct-report via `member add --team`:** `jane@co.com.md` frontmatter contains `current_manager: "[[...|me@co.com]]"`, `previous_manager: []`, `other_leaderships: []`, `start_date: ""`, `projects: []`, `teams: ["[[...|backend]]"]` (wiki-link to team context file). Self-profile `direct_reports:` is updated with jane's wiki-link.

**AC2 — Company-member (no team):** `bob@co.com.md` frontmatter contains `current_manager: ""` (empty), `previous_manager: []`, `other_leaderships: []`, `start_date: ""`, `projects: []`. Self-profile `direct_reports:` is **not** updated (decision #6 — company member is not a direct report).

**AC3 — Contractor:** `ext@vendor.com.md` frontmatter contains `current_manager: ""` (empty), same defaults as AC2. Self-profile `direct_reports:` is **not** updated.

**AC4 — Direct-report via `team add`:** Same vocabulary as AC1 generated by `TeamService.buildMemberProfileMd`. Self-profile `direct_reports:` is updated with the member's wiki-link.

**AC5 — `buildMemberProfileMd` template clean:** New template contains no `action_items_gdoc`, no `## Current Manager`, `## Previous Managers`, `## Other Leaderships`, `## Previous Leaderships`, `## Action Items` body sections, no `[[action-items-<email>|Action Items Tracker]]` link. Kept body sections: `## 1on1s`, `## Feedbacks`, `## Assessments`, `## Performance Reviews`, `## Notes`.

**AC6 — No action-items file creation:** `TeamService.addMember` no longer creates `action-items-<email>.md` (the link and field are gone; the file is now orphaned noise).

**AC7 — Idempotent:** Running either command twice for the same email does not duplicate any frontmatter entry. `addRelation` guarantees idempotency for array fields. Second run returns `{ created: false }` for `MemberService.addMember`.

**AC8 — Self-profile absent:** When `my-career/` does not exist or contains no `.md` files, the `current_manager` field in the member profile is `""` and the self-profile `direct_reports:` write is silently skipped (no error).

**AC9 — Existing profile idempotent:** When `member add --team` runs on an already-existing profile, `direct_reports` is still synced on the self-profile (idempotent via `addRelation`).

### New Private Helper: `MemberService._resolveTeamContextLink`

```typescript
private _resolveTeamContextLink(teamName: string, fromPath: string, workspaceRoot: string): string {
  const slug = normalizeSlug(teamName);
  const contextPath = path.join(workspaceRoot, 'my-teams', 'teams', slug, `${slug}-context.md`);
  return formatWikiLink(contextPath, fromPath, slug);
}
```

Note: this is synchronous (no filesystem reads needed) — we're generating the wiki-link path, not verifying file existence.

### New Private Helper: `MemberService._getSelfProfilePath`

```typescript
private async _getSelfProfilePath(workspaceRoot: string): Promise<string | null> {
  const careerRoot = path.join(workspaceRoot, 'my-career');
  if (!(await this._fs.exists(careerRoot))) return null;
  const mdFiles = await this._fs.listFiles(careerRoot, '.md');
  return mdFiles.length > 0 ? (mdFiles[0] as string) : null;
}
```

This shares logic with `_resolveManagerLink` — they both scan `my-career/*.md`. Consider extracting the scan into a shared `_findSelfProfilePath` helper that both methods use, to avoid duplication.

### Reciprocal `direct_reports` Write in `MemberService`

Add a private method `_syncDirectReports` and call it from both the existing-profile path and the new-profile path:

```typescript
private async _syncDirectReports(
  opts: IAddMemberOptions,
  profilePath: string,
  normalizedEmail: string,
  workspaceRoot: string,
): Promise<void> {
  if (!opts.team || opts.contractor) return; // only for direct reports

  const selfPath = await this._getSelfProfilePath(workspaceRoot);
  if (!selfPath) return; // no self profile found — skip silently

  const memberLink = formatWikiLink(profilePath, selfPath, normalizedEmail);
  await addRelation(selfPath, 'direct_reports', memberLink, this._fs);
}
```

Call order in `addMember`:
1. If profile exists: `_syncTeamMembersFrontmatter` then `_syncDirectReports` then return `{ created: false }`
2. If new profile: write profile → `_syncTeamMembersFrontmatter` → `_syncDirectReports` → return `{ created: true }`

**Import required:** `addRelation` is already imported in `member.service.ts`.

### Exact Change: `TeamService.buildMemberProfileMd` — Full Rewrite

The function signature stays the same. Change from raw template literal to `matter.stringify`:

```typescript
function buildMemberProfileMd(
  email: string,
  options: { name?: string; role?: string; gender?: string; location?: string },
  teams: string[],
  managerEmail: string | null,
  workspaceRoot: string,
): string {
  const managerLink = managerEmail
    ? formatWikiLink(
        path.join(workspaceRoot, 'my-career', `${managerEmail}.md`),  // flat post-9.3
        memberProfilePath(workspaceRoot, email),
        managerEmail,
      )
    : '';

  const teamLinks = teams.map((slug) =>
    formatWikiLink(
      path.join(workspaceRoot, 'my-teams', 'teams', slug, `${slug}-context.md`),
      memberProfilePath(workspaceRoot, email),
      slug,
    ),
  );

  const fm: Record<string, unknown> = {
    email: `"${email}"`,  // NOTE: use matter.stringify — it handles quoting
    name: options.name ?? '',
    role: options.role ?? '',
    gender: options.gender ?? '',
    location: options.location ?? '',
    relationship: 'direct-report',
    date_added: todayIso(),
    start_date: '',
    current_manager: managerLink,
    previous_manager: [],
    other_leaderships: [],
    teams: teamLinks,
    projects: [],
  };

  const body = '\n## 1on1s\n\n## Feedbacks\n\n## Assessments\n\n## Performance Reviews\n\n## Notes\n';
  return matter.stringify(body, fm);
}
```

**IMPORTANT:** `matter.stringify` handles YAML serialization. Do NOT manually build the YAML string. The `email: "${email}"` pattern from the old template put literal quotes — let gray-matter handle this naturally (pass the raw email string).

**WARNING on `getManagerEmail()`:** `TeamService.getManagerEmail()` uses `listDirectories` which returns empty on flat `my-career/` (pre-existing D1 deferred). When `buildMemberProfileMd` is called via `TeamService.addMember`, `managerEmail` will often be `null` → `current_manager: ''`. This is pre-existing behavior. **Do NOT fix `getManagerEmail()` in this story.** However, DO fix it in `_getSelfProfilePath` below to correctly find the self profile for `direct_reports`.
